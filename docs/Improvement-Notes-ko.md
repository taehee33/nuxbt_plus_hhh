# NUXBT 개선사항 정리

이 문서는 `개인 보관용 개발 메모`와 `사용자에게 공개할 수 있는 개선 안내`를 한 곳에 정리한 문서입니다.

용도를 분리하면 다음과 같습니다.

1. 개인 보관용:
   - 실제로 무엇이 문제였는지
   - 어떤 방식으로 수정했는지
   - 아직 남아 있는 한계가 무엇인지
   - 다음 작업 우선순위가 무엇인지
2. 사용자 공개용:
   - 지금 무엇이 좋아졌는지
   - 사용자가 어떤 점에서 덜 헷갈리게 되었는지
   - 앞으로 어떤 부분이 더 좋아질 예정인지

---

## 1. 문서 목적

이 문서의 목적은 단순한 "개선 아이디어 목록"을 만드는 것이 아닙니다.

실제 목적은 다음 네 가지입니다.

1. macOS + VirtualBox + Vagrant 환경에서 NUXBT를 실행하며 부딪힌 실제 문제를 기록한다.
2. 제품 개선 항목과 운영 개선 항목을 분리한다.
3. 공개 가능한 개선 내역과 내부 관리용 메모를 분리한다.
4. 다음 작업 우선순위를 명확히 남긴다.

---

## 2. 현재 개선 작업의 배경

이번 작업은 단순히 문서를 조금 정리한 수준이 아니라, `macOS 사용자가 실제로 NUXBT를 실행 가능한 상태까지 만드는 과정`에서 나온 개선 작업입니다.

특히 다음 문제가 반복적으로 확인됐습니다.

1. macOS에서는 NUXBT를 직접 실행할 수 없고 Linux VM이 필요하다는 사실이 초반에 명확하지 않음
2. 사용자가 블루투스 어댑터와 주변 기기 목록을 혼동함
3. VirtualBox USB 패스스루 상태와 BlueZ 어댑터 상태를 한 번에 이해하기 어려움
4. 웹 UI가 비어 보이거나 동작하지 않을 때 원인 파악이 어려움
5. GitHub Actions가 자동 실행되면서 개발/문서 수정 단계에도 불필요한 실패 로그가 쌓임
6. 로컬 배포 산출물과 실제 소스 관리 영역이 분리되어 있지 않음

즉, 이번 개선의 핵심은 `기능 추가`보다 `실행 가능성`, `진단 가능성`, `사용자 이해도`를 높이는 데 있습니다.

## 3. 이번에 반영된 주요 개선사항

### 3.1 웹 UI에서 블루투스 어댑터 정보 강화

반영 내용:

1. `Detected Adapters` 패널 추가
2. 각 어댑터에 대해 다음 정보를 표시
   - alias
   - address
   - powered
   - pairable
   - discoverable / hidden
   - manufacturer
   - product
   - USB vendor/product ID
   - BlueZ path
3. 다음 컨트롤러 생성에 사용할 어댑터 선택 기능 추가
4. 추천 가능한 어댑터에 `Recommended` 표시 추가

개선 효과:

1. 사용자가 "NUXBT가 실제로 어떤 블루투스 어댑터를 쓰는지" 볼 수 있음
2. 단순히 "블루투스가 안 된다"가 아니라 어떤 어댑터가 잡혔는지 확인 가능
3. 다중 어댑터 환경에서 다음 생성 대상 선택 가능

### 3.2 Host USB Devices 확인 흐름 추가

반영 내용:

1. macOS 호스트 USB 목록을 읽기 위한 `host_usb_bridge.py` 추가
2. 웹 UI에서 `Host USB Devices` 패널을 통해 호스트 USB 장치 상태 확인 가능
3. CSR 동글 같은 실제 USB 블루투스 장치가 `Captured` 상태인지 확인 가능

개선 효과:

1. VirtualBox가 USB 동글을 VM에 넘겼는지 쉽게 파악 가능
2. BlueZ 어댑터가 안 보일 때 원인을 호스트 측에서 먼저 좁힐 수 있음
3. VM 내부 문제와 호스트 USB 패스스루 문제를 구분하기 쉬워짐

### 3.3 macOS 초보자용 문서 보강

반영 내용:

1. `docs/Usage-Guide-ko.md` 추가 및 정리
2. `docs/QUICKSTART-ko.md` 추가
3. README에서 한국어 문서와 빠른 시작 경로 링크 제공
4. `scripts/start_macos_vm_webapp.sh` 추가

개선 효과:

1. 처음 실행하는 사용자가 필요한 순서를 짧게 따라갈 수 있음
2. VM 실행, host bridge, 웹앱 실행이 하나의 시작 스크립트로 정리됨
3. 설치 문서와 사용 문서가 어느 정도 역할 분리됨

### 3.4 웹 정적 자산 캐시 문제 완화

반영 내용:

1. 루트 HTML에 캐시 방지 헤더 적용
2. `?v=1` 방식의 cache busting 안내 추가
3. 정적 자산이 갱신된 뒤에도 브라우저가 오래된 bundle을 보는 상황에 대한 대응 문서화

개선 효과:

1. 흰 화면 발생 시 단순 새로고침 이상의 해결책 제공
2. 프런트 번들 404 문제를 사용자도 비교적 쉽게 이해 가능
3. "사이트가 아예 죽었다"와 "캐시가 꼬였다"를 구분하기 쉬워짐

### 3.5 GitHub Actions 자동 실행 범위 축소

반영 내용:

1. `bump_version.yml`을 수동 실행만 되게 변경
2. `release.yml`도 수동 실행만 되게 변경
3. GitHub App token 의존성 제거

개선 효과:

1. 일반 푸시마다 불필요한 실패 로그가 쌓이지 않음
2. 문서 수정이나 실험성 커밋이 배포 파이프라인을 건드리지 않음
3. 실제 릴리스 시점에만 의도적으로 워크플로를 실행 가능

## 4. 개인 보관용 상세 메모

이 섹션은 공개 문안보다 더 솔직하고 실무적인 관찰 메모를 담습니다.

### 4.1 가장 많이 헷갈렸던 지점

1. 사용자는 `Detected Adapters`를 주변 블루투스 스캔 목록으로 이해하려는 경향이 강함
2. 실제로 필요한 것은 "주변 기기 목록"이 아니라 "NUXBT가 컨트롤러로 사용할 수 있는 어댑터 목록"임
3. macOS 사용자는 USB 동글을 꽂아도 왜 VM 안에서 안 보이는지 이해하기 어려움
4. `VBoxManage list usbhost`와 `bluetoothctl list`가 각각 다른 층위의 정보라는 점이 직관적이지 않음

### 4.2 실제 운영에서 중요했던 확인 순서

문제 발생 시 가장 효율적인 확인 순서는 다음이었습니다.

1. 호스트에서 `VBoxManage list usbhost`
2. VM에서 `lsusb`
3. VM에서 `bluetoothctl list`
4. VM에서 `hciconfig -a`
5. VM에서 `nuxbt check`
6. 웹 UI에서 `Detected Adapters`
7. 필요 시 `Host USB Devices`

이 순서를 문서와 스크립트에 더 직접적으로 반영할 가치가 있습니다.

### 4.3 사용자가 실제로 막히는 지점

1. VM은 떠 있는데 웹 UI가 안 열림
2. 웹 UI는 열리는데 `Create Pro Controller`가 반응하지 않음
3. `No adapters available`가 뜸
4. `Detected Adapters`가 하나만 보여서 버그라고 생각함
5. 빈 화면이 나와서 서버 자체가 죽은 줄 앎

즉, 문제의 대부분은 "기능 자체"보다 "상태가 보이지 않는 것"에서 발생합니다.

### 4.4 지금 구조의 현실적인 한계

1. macOS 호스트 USB 정보는 웹앱 서버가 직접 읽을 수 없고 브리지가 필요함
2. 내장 블루투스는 사실상 안정 경로가 아니므로 외장 동글 전제가 강함
3. WebRTC / 브라우저 캐시 / VM 네트워크 문제가 섞이면 사용자가 독립적으로 원인 분리를 하기 어려움
4. Linux 패키징과 macOS 실행 경험이 동일한 수준으로 매끄럽지는 않음

즉, 이 프로젝트의 macOS 지원은 "네이티브 앱 수준의 경험"이 아니라 "VM 기반 운영 경험을 최대한 덜 불편하게 만드는 방향"으로 보는 것이 정확합니다.

## 5. 사용자에게 공개할 수 있는 개선사항 정리

이 섹션은 GitHub README, Release Note, 공지 글, 블로그 포스트 같은 공개용 텍스트의 기반으로 사용할 수 있습니다.

### 5.1 공개용 요약 문안

다음과 같이 정리할 수 있습니다.

```text
macOS + VirtualBox + Vagrant 환경에서 NUXBT를 더 쉽게 실행하고 진단할 수 있도록 웹 UI와 문서를 개선했습니다.

- 블루투스 어댑터 목록과 상태를 웹 UI에서 직접 확인 가능
- Manufacturer / Product / USB ID 표시 추가
- 호스트 USB 장치 상태 확인을 위한 bridge 및 UI 연동 추가
- macOS 초보자용 QUICKSTART 및 한방 실행 스크립트 추가
- 웹앱 캐시 문제에 대한 대응 문서 보강
- GitHub Actions를 수동 실행 중심으로 정리해 개발 중 불필요한 자동 배포/릴리스 시도를 줄임
```

### 5.2 공개용 상세 문안

사용자에게 설명할 때는 아래 관점이 좋습니다.

1. 실행 경로가 더 명확해졌다
   - macOS 사용자는 VM 기반 실행 경로를 바로 따라갈 수 있음
2. 진단 정보가 더 많아졌다
   - 어댑터 정보, USB 정보, 상태값을 웹 UI에서 확인 가능
3. 실패 원인을 더 빨리 좁힐 수 있다
   - 어댑터 문제인지, USB 패스스루 문제인지, 캐시 문제인지 구분 가능
4. 문서가 더 실전형이 되었다
   - QUICKSTART, Usage Guide, Troubleshooting 중심으로 보강

### 5.3 공개용 릴리스 노트 예시

```text
이번 업데이트에서는 macOS + VirtualBox + Vagrant 환경에서의 사용성과 진단 흐름을 중심으로 개선했습니다.

주요 변경사항:
- 웹 UI에 Detected Adapters 패널 추가
- Manufacturer, Product, USB ID 등 어댑터 메타데이터 표시 강화
- Host USB Devices 확인을 위한 host USB bridge 추가
- macOS QUICKSTART 문서와 one-shot 실행 스크립트 추가
- 웹앱 캐시 문제 대응 개선
- GitHub Actions 릴리스/버전 워크플로 수동 실행 전환
```

## 7. 우선순위 제안

### 높은 우선순위

1. 사용자 친화적인 런타임 에러 화면 추가
2. `Detected Adapters`와 `Host USB Devices` 관계를 더 직접적으로 설명
3. VM 블루투스 상태 점검 스크립트 추가
4. QUICKSTART에 실제 화면 스크린샷 포함

### 중간 우선순위

1. `Nearby Devices` 패널 추가
2. 선택된 어댑터와 생성 흐름의 연결성 UI 강화
3. 릴리스 산출물 관리 정책 정리
4. README의 macOS 섹션을 더 초보자 친화적으로 재구성

### 낮은 우선순위

1. 복잡한 자동 배포 정비
2. PPA / PyPI / RPM 릴리스 플로우 완전 자동화
3. 다중 플랫폼 패키징 전략 통합

## 8. 개인 보관용 결론

이번 작업의 핵심은 `NUXBT의 macOS 네이티브 지원`을 만든 것이 아니라, `VM 기반 사용 경로를 현실적으로 운영 가능하게 다듬은 것`입니다.

즉, 현재 상태를 가장 정확하게 표현하면 다음과 같습니다.

1. macOS 사용자가 처음 실행할 수 있는 문서와 스크립트가 생겼다.
2. 블루투스 어댑터와 호스트 USB 상태를 웹에서 더 잘 볼 수 있게 됐다.
3. 실패했을 때 어디를 봐야 하는지 이전보다 훨씬 명확해졌다.
4. 다만 완전한 plug-and-play 수준은 아니고, 여전히 VM/USB/BlueZ 구조를 이해해야 한다.

## 9. 사용자 공개용 결론

사용자에게는 아래 정도로 정리하는 것이 적절합니다.

```text
macOS + VirtualBox + Vagrant 환경에서 NUXBT를 더 쉽게 실행하고 문제를 진단할 수 있도록 웹 UI, 문서, 실행 보조 스크립트를 개선했습니다.

특히 블루투스 어댑터 정보와 호스트 USB 상태를 더 명확히 볼 수 있게 했고, 초보자도 따라할 수 있는 QUICKSTART 문서와 한방 실행 스크립트를 추가했습니다.
```
